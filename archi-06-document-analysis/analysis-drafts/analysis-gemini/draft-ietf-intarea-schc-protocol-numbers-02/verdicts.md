## Executive assessment

SCHC Architecture -06 can naturally express draft-ietf-intarea-schc-protocol-numbers-02. This is not a coincidence of vocabulary: -06's Appendix A.2.3 is written about this draft by name, and it supplies the architectural reading of all three requested codepoints in a single sentence — the EtherType, IP Protocol Number, and UDP Port Number each "serves as the Discriminator: it indicates that the frame or packet carries a SCHC Datagram, and the Dispatcher uses it, together with lower-layer context (e.g., addresses or ports), to route the Datagram to the appropriate Instance."

The principal conceptual mapping is therefore **codepoint → Discriminator, consumed by the Dispatcher of the receiving Endpoint, in combination with lower-layer context**. The draft's own packet picture (Figure 1: RuleID, Compression Residue, Payload) is -06's Datagram Format with a trailing Payload, and needs no reinterpretation. The draft's Section 4 requirement that the datagram length be carried explicitly in the Compression Residue when the 802 header does not supply it is a legitimate profile-level constraint on Rule content, of exactly the kind -06 Section 5 anticipates.

The principal migration difficulty is the draft's Section 1.2 sentence "the use of SCHC to compress the transported protocol, as well as the SCHC session (called instance) to use, are implicit." That sentence treats "session" and "instance" as synonyms. In -06 they are distinct: an Instance is a logical component of an Endpoint that executes SCHC operations and owns a Context and an Instance Configuration, whereas a Session is a communication among two or more Instances sharing a Context. The draft's sentence is true of both under the LoRaWAN topology it describes, precisely because that topology forces a one-to-one correspondence — which is why the conflation is invisible there and breaks the moment the draft extends the argument to Ethernet, where an Endpoint may host several Instances. Migrating the sentence requires saying which of the two the Ethertype signals (neither: the Ethertype signals that SCHC is present; Instance and Session selection is the SCHC WG's chartered follow-on work). This is a decomposition, not a change of technical intent, and per the grading rules it does not by itself lower conceptual equivalence.

An Architecture gap does exist, and it is small. The draft's Section 6.3 floats an IANA registry that structures the RuleID: the top nibble of a 1-byte RuleID would select a "domain of use" such as aviation, and the value 15 would escape to a 2-byte RuleID. Two things are being asked of the RuleID here that -06 never grants or denies. First, that RuleID values carry meaning outside the Context in which they were minted — an IANA registry is a global administrative scope. Second, that a prefix of the RuleID be decodable, and select the RuleID's own length, before any Context is bound to the Datagram. -06 comes close: Section 4.2.5.1 already accepts that "a profile may define a fixed RuleID size and specify that RuleID values in a particular range are followed by a Control Header having a known format," and that "those framing semantics are independent of the C/D or F/R Rule selected by the RuleID." But -06 never defines RuleID in its Terminology section and never states its uniqueness scope, so the reader has no basis on which to judge the registry proposal. Closing the gap is additive: define RuleID, state that it is Context-scoped rather than Domain-scoped, and state that its size must be determinable before parsing. No existing -06 text changes meaning.

Because the adaptation need is Trivial, `schc-architecture-edits.md` accompanies this file. Because conceptual equivalence is High and transition difficulty is Easy, `terminology-migration.diff` accompanies it as well, and it is complete: the migration was performed end-to-end without any location requiring wording that does not exist in -06 or in the accepted edits.

### Native architectural model

The draft is, on its face, a request for three codepoints: an Internet Protocol Number, an Ethertype, and a UDP port. But it carries an architectural argument, and that argument is what must be mapped. The argument is that SCHC's original deployment model — SCHC bound implicitly to a particular constrained link, with the fact of SCHC's presence inferred from the link rather than declared — does not survive contact with three situations: security envelopes, general-purpose 802 networks, and firewalled paths. In each, something in the packet must say "this is SCHC," and today nothing can.

The security-envelope case is the sharpest. When an IP packet's Next Header is ESP, the ESP receiver has no reliable way to know whether the payload it is about to process is ordinary ESP per RFC 4303 or a Diet ESP payload that has been SCHC-compressed. The draft acknowledges that the SPI can be allocated so as to encode this distinction implicitly, and rejects that: "it is not sound protocol design." Its remedy is to make SCHC visible in the Next Header field itself, so that "a clear segregation of incoming traffic is directly supportable." The same reasoning is applied to DTLS 1.3, where the draft observes that the DTLS header offers no SPI-equivalent hook at all, so implicit SCHC has nowhere to hide, and an explicit IP-layer indication becomes necessary rather than merely preferable.

Having made SCHC explicit at the IP layer, the draft then observes that SCHC can also be nested: SCHC can appear as the Next Header *within* the ESP header, with ordinary SCHC rules governing that inner content. This single sentence carries a great deal of weight. It means the draft's model admits more than one SCHC processing step per packet, at different positions in the protocol stack, each with its own rules, each declared by its own Next Header value. The draft develops this no further, but it is not compatible with a model in which a node performs SCHC once.

The Ethertype argument runs on a different axis. On a classical LPWAN link such as LoRa, two facts are given for free by the link: that the payload is SCHC at all, and which SCHC state applies. Both come from preconfiguration of the MAC-layer endpoints. The draft's words are that "there can be only one session, and there can be only SCHC." Extended to Ethernet and to more capable endpoints, both facts stop being free, and the draft says so: "it is necessary to signal both the use of SCHC and the SCHC session to be used." The draft then divides the labour explicitly — the Ethertype it defines will signal the first, and the SCHC working group is chartered to produce the mechanism for the second. This is a clean and correct separation, and it is stated in the same breath as the draft's one significant terminological collapse, the parenthetical "(called instance)" that makes session and instance the same word.

That collapse is understandable and, within Section 1.2's own frame, harmless. On the LoRa link the draft is describing, preconfiguration of a single MAC peer pair means there is exactly one of whatever-it-is: one bundle of compression state, associated with one peer, carrying one conversation. Session and instance are indistinguishable because the topology has made them so. The collapse becomes consequential exactly where the draft takes it — Ethernet — because a capable endpoint on a shared 802 medium may plausibly hold several bundles of compression state and take part in several conversations, and these need not correspond one-to-one.

Receive-side state in the draft resides in one named structure: a table, held by the receiving implementation, mapping source IP addresses to RuleID sizes, with addresses in prefix form so that address groups can share a size. This is a modest sentence with two firm implications. First, the receiver's dispatch state is keyed on the peer, not on the node: it is per-peer-group, not global. Second, the RuleID's own length is not a property of the SCHC packet but of the relationship with the sender, and must be resolved from the sender's identity before the SCHC packet can be parsed at all. The draft therefore already assumes a two-stage receive path — identify the peer from lower-layer information, then parse the RuleID — even though it never says so.

Rule identity in the draft may be established two ways. It "may be statically configured per [RFC8724], or may be negotiated within a protocol as in IKE [ikev2-diet-esp]." The second is a real departure from RFC 8724's premise that no negotiation takes place between compressor and decompressor. The draft states it without comment. Where a negotiated RuleID lives is not specified, but the IKEv2 reference implies it lives with the security association: per peer pair, established at the time the SA is, and lasting as long.

The Ethertype case forces one further piece of state into the wire format. An 802 header does not necessarily carry the payload length, so a receiver that must know the datagram length has to get it from somewhere. The draft's answer is normative and specific: "If the rules for the RuleID does not provide the datagram length, the datagram length MUST be explicit in the Compression Residue." Length is thus a conditional, rule-dependent element of the residue — present when the rule cannot supply it, absent when the rule can. This is the only MUST in the draft's technical body, and it constrains rule authors rather than protocol implementers.

Finally, Section 6.3 opens a question the draft does not close. It suggests that a registry of SCHC RuleIDs may be needed for the Ethertype case, and sketches one: split a 1-byte RuleID in half; assign top-nibble values 1 through 14 to "different domains of use, like for aviation"; let the value 15 mean that a 2-byte RuleID is in use. The section is candidly provisional — "may be needed," "More discussion is needed to resolve this" — but the sketch is architecturally loaded. It would make a prefix of the RuleID self-describing (the length escape), and it would make another part of the RuleID meaningful across deployments that have never exchanged a context (the domain-of-use nibble). It also introduces a second, incompatible mechanism for determining RuleID size, competing with the source-address table of Section 3. The draft nowhere reconciles these, and nowhere states in what scope a RuleID value is unique.

Two ambiguities are therefore carried forward into the mapping rather than resolved. The first is whether "session (called instance)" denotes per-peer conversation state, a bundle of compression machinery, or both fused. The second is whether Section 6.3's registry is a genuine requirement of this draft or an open question recorded for the working group. Both are treated below as ambiguities; where a grade depends on the reading, the most defensible reading is used and the alternative is stated.

### Architectural risk points

**R1 — RuleID uniqueness scope is undefined in -06 and internally inconsistent in the draft.**

- **Risk:** -06 states that "Identifiers for Instances, Contexts, and Sessions are unique within the scope of their Domain" (Section 4.1.2) and never says anything about the RuleID's scope; it has no Terminology entry for RuleID at all. The draft meanwhile uses RuleID two incompatible ways: in Sections 3 and 4 as a Context-relative index reached from the sender's identity, and in Section 6.3 as an IANA-registered value whose top nibble means "aviation" to any receiver.
- **Why it matters:** Identifier scope determines who may assign a value and where a collision is possible. If RuleIDs are Context-scoped, an IANA registry is a coordination convention among profiles and grants no global uniqueness. If they are globally scoped, then -06's entire Context-sharing model — in which two Instances interoperate because they hold the same Context, not because they consulted a registry — has a second, unacknowledged identifier authority. Neither document says which.
- **Consequence for migration:** Section 6.3 cannot be migrated without stating the answer. The migration diff resolves it in the Context-scoped direction (the reading consistent with RFC 8724, with -06's Section 4.2.5, and with the draft's own Sections 3 and 4), and recasts the registry as a cross-Domain reservation of RuleID ranges rather than a global uniqueness claim. This is the only place where the migration takes a position the draft does not.

**R2 — Two unreconciled mechanisms determine RuleID size.**

- **Risk:** Section 3 says the receiver holds "a table of source IP address and RuleID size." Section 6.3 says a RuleID value of 15 in the top nibble "designates that a 2-byte RuleID is used." These are different answers to the same question, and the draft does not say whether they coexist, whether one is for the IP-number case and the other for the Ethertype case, or which wins if both apply.
- **Why it matters:** RuleID size must be resolved before the Datagram can be parsed. -06 Section 4.2.5.1 states the analogous requirement for the Control Header — "The information needed to locate and decode the Control Header must be known before that information is used" — but never extends it to the RuleID itself. A deployment reading -06 alone has no instruction to make the RuleID size determinable a priori.
- **Consequence for migration:** Both mechanisms map to -06 once Architecture edit 3 is accepted, one as Instance Configuration dispatch information and one as profile-defined framing. The migration does not need to choose between them, but it must stop presenting them as unrelated remarks in different sections. The diff scopes each to its own codepoint case and notes the shared prerequisite.

**R3 — Nested SCHC is asserted once and is untested against -06's Dispatcher model.**

- **Risk:** "Additionally, SCHC can then be the Next Header within the ESP header with 'regular' SCHC rules for processing this content." One packet, two SCHC operations, at different points in the stack. -06 permits multiple Instances per Endpoint and gives Stratum as the concept that separates their reach, but Stratum is labelled "a background concept," no -06 example exercises it, and Section 4.2.2.4's Dispatcher is described as admitting packets from and reinjecting them into the network stack — a model that plainly readmits a decrypted inner packet, but is nowhere shown doing so.
- **Why it matters:** If -06's Dispatcher is a single point of decision per Endpoint, then a nested packet must traverse it twice, once before decryption and once after, with a different Discriminator each time. Nothing in -06 forbids this; nothing in -06 confirms it.
- **Consequence for migration:** None for the text of this draft — the sentence is motivation, not specification, and migrates unchanged. But the draft's core justification for the IP Protocol Number rests on nesting working, and neither document demonstrates it. This is a gap in worked examples, not in concepts, and is therefore not an ARCHITECTURE GAP. It belongs on the list of deployment models -06 Appendix B.2 says "remain to be documented."

**R4 — The IKEv2-negotiated RuleID sits astride -06's no-negotiation foundation.**

- **Risk:** "The RuleID may be statically configured per [RFC8724], or may be negotiated within a protocol as in IKE [ikev2-diet-esp]." -06 Section 1 states that SCHC "assumes that the Context is static and provisioned before use, and that no negotiation takes place between the compressing and decompressing entities," and that "these assumptions remain the foundation of this architecture."
- **Why it matters:** The two statements are reconcilable only if the IKEv2 exchange is understood as Context *provisioning* — which -06 explicitly allows to be dynamic ("Contexts may be fetched on demand") — and not as SCHC negotiation between compressor and decompressor. The distinction is real and the draft never draws it. A reader could take the draft's sentence as licensing in-band SCHC negotiation, which -06 forbids at its foundation.
- **Consequence for migration:** The migration must make the distinction explicit, in one clause. It does not change what the draft permits, because IKEv2 is by construction an out-of-band provisioning exchange. Classified REQUIRED FOR TERMINOLOGY MIGRATION, not conceptual alignment: the draft's behavior is already compliant; only its formulation is silent.

**R5 — "Domain" is a false friend across the two documents.**

- **Risk:** -06's Domain is "a logical grouping of Instances that share a common set of Contexts," managed by a Domain Manager and never carried on the wire. The draft's "different domains of use, like for aviation" are IANA-assigned RuleID prefixes. The word is the same; nothing else is.
- **Why it matters:** A migration performed by terminology substitution would capitalise the draft's "domains" and thereby assert that an aviation deployment is an -06 Domain identified by four bits on the wire — an architectural claim the draft never makes and -06 does not support.
- **Consequence for migration:** Section 6.3's "domains of use" must be left in lower case and explicitly distinguished from -06's Domain. This is the second-hardest judgment call in the migration, after the session/instance decomposition.

**R6 — "Endpoint" is a false friend across the two documents.**

- **Risk:** The draft's "MAC-Layer endpoints are preconfigured" (Section 1.2) refers to link-layer peers. -06's Endpoint is a logical SCHC entity, explicitly "rather than a physical device," and several may share one physical device.
- **Why it matters:** As with R5, a replace-all migration produces a false statement: that SCHC Endpoints are preconfigured at the MAC layer.
- **Consequence for migration:** This occurrence of "endpoint" must not be translated. The migration diff rewrites it to "MAC-layer peers" to remove the collision. -06's own A.2.1 confirms the correct reading by treating the DevEUI as the source of a Discriminator, not as an Endpoint.

## Final migration assessment

- **Can the draft be migrated without changing technical behavior?** **Mostly.** Sections 3, 4, 5, 6.1 and 6.2 migrate with no behavioral change whatsoever; the requested codepoints, the IANA tables, the packet layout and the sole normative MUST are all untouched. Two locations change an assumption rather than a word. Section 1.2's "session (called instance)" must be split, which does not alter what any implementation does but does alter what the paragraph asserts about SCHC in general. Section 6.3 must state that its proposed registry does not make RuleIDs globally unique — a position the draft has not taken, and the only place in the migration where a technical assumption is resolved rather than restated.
- **Can the migration be performed mechanically?** **Mostly.** The three judgment calls (RuleID scope in Section 6.3; the session/instance split in Section 1.2; recognising that "endpoint" and "domain" are false friends in Sections 1.2 and 6.3) are each made once and then applied without further discretion. Everything downstream — Datagram naming, Discriminator framing of the three codepoints, the dispatch-information rewording — is substitution. The accompanying `terminology-migration.diff` is complete and contains no `<<BLOCKED>>` marker, which is the evidence supporting the Easy verdict.
- **Does the draft expose a SCHC Architecture -06 gap?** **Yes.** -06 does not define the RuleID in its Terminology section and does not state its uniqueness scope. Section 4.1.2 scopes Instance, Context and Session identifiers to the Domain and passes over the RuleID in silence; Section 4.2.5.1 requires the Control Header to be locatable before parsing but says nothing about the RuleID's own size. A reader of -06 cannot determine whether an IANA registry of RuleID values, as Section 6.3 of the draft proposes, is architecturally coherent.
- **Is the gap required for this draft or merely useful generally?** **Required for this draft, and useful generally.** Section 6.3 cannot be migrated to -06 terminology without an answer to the scope question, so the gap is squarely on this draft's critical path. But the question is not peculiar to it: any specification that carries a SCHC Datagram over a shared medium, or that reserves RuleID values across deployments, needs the same answer, and -06's Section 4.2.5.1 already leans on the unstated RuleID-size prerequisite that edit A3 supplies. The three edits are additive clarifications that would improve -06 independently of this draft, which is why the adaptation need is Trivial rather than Medium.
- **What is the single most important migration issue?** The RuleID scope question raised by Section 6.3. Everything else in this draft either maps Directly onto -06 or was, in the case of the three codepoints, written into -06's Appendix A.2.3 in advance. Section 6.3 alone asks -06 for something it has not defined — an identifier scope in which a RuleID prefix can mean "aviation" to a receiver that shares no Context with the sender — and it does so in a section the draft itself marks unresolved. Resolving it in the Context-scoped direction, and recasting the registry as a cross-Domain range reservation rather than a global uniqueness claim, is both the one substantive conceptual edit the draft needs and the one thing -06 must say for the edit to be sayable.
